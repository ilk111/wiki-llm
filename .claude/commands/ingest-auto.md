# Автоматический batch-ingest

Трёхфазный workflow: оркестратор детектит новые файлы и делает triage (для архитектурного контекста, **не** для выбора модели) → параллельные subagents на указанной модели делают тяжёлую работу → оркестратор собирает всё в wiki одним согласованным решением с валидацией цитат.

**Зачем отдельно от `/ingest`:** когда в `vault/raw/**` накопилось несколько новых файлов, обрабатывать их по одному — долго, и таксономия дёргается после каждого. Здесь subagents работают параллельно (свежий контекст, без галлюцинаций от соседних источников), а таксономия решается один раз по всей партии.

## Аргумент

`$ARGUMENTS` — модель для всей партии. Обязательный. Допустимо только `sonnet` или `opus`.

- `sonnet` → все subagents на Sonnet 4.6.
- `opus` → все subagents на Opus 4.7.

**Если `$ARGUMENTS` пустой или не равен `sonnet`/`opus`** — останови выполнение и скажи пользователю: «Укажи модель для партии: `/ingest-auto sonnet` или `/ingest-auto opus`. Haiku для ingest не используется». Дальше не иди.

**Обе модели работают в режиме extended thinking с предмаксимальным усилием** — и оркестратор (ты), и subagents. Это задачи, где качество размышлений (triage, подбор цитат, архитектурное обоснование, решения по таксономии) напрямую определяет результат. Прокидывай это требование в промпт каждого subagent явно.

---

## Шаг 0: детект новых файлов

1. Прочитай `vault/wiki/log.md`, извлеки все строки `Источник: ` — это список уже ingested путей.
2. Собери полный список raw-файлов: `Glob vault/raw/**/*.{pdf,md,mp3,mp4,wav,m4a,webm}`.
3. Исключи:
   - всё что упомянуто в `Источник:` из log.md (нормализуй пути — там могут быть кавычки, бэктики)
   - парные `.md` рядом с `.pdf` с тем же базовым именем (это Docling-выгрузка, ingested как часть pdf)
   - парные `.mp3` рядом с `.md` с тем же базовым именем (это переслушиваемый транскрипт)
   - файлы из `vault/raw/assets/` (картинки/схемы, не ingest-материал)
4. Покажи пользователю финальный список с путями и размерами (`ls -lh`). Кратко спроси: «Обрабатываем все N файлов на модели <model>? Или исключить какие-то?» — **жди ответа.**

Если список пустой — сообщи «новых файлов не найдено» и выйди.

---

## Шаг 0.5: triage — архитектурная классификация партии

Для каждого файла из утверждённого списка определи: обычный или архитектурный. **Это решает только одно — передавать ли subagent'у блок `<ARCHITECTURAL_NEIGHBOUR_PAGES>` и требовать ли секцию «Архитектурное обоснование». Модель не меняется от результата triage — она задана пользователем через `$ARGUMENTS`.**

Triage делает оркестратор (ты), потому что требует знания состояния wiki целиком.

**Источники сигнала (читай сам):**
- `vault/wiki/index.md` целиком — текущая структура;
- `vault/wiki/log.md` — последние 20 записей, чтобы видеть свежие таксономические решения;
- `vault/wiki/conflicts.md` — что уже находится в споре.

**Для каждого raw-файла задай 3 вопроса** (используй только имя файла и, если нужно, первые ~50 строк содержимого — не читай целиком, это задача subagent):

1. **Новая тема?** По имени файла / первым абзацам — в `index.md` есть ли уже ≥2 страницы по этой теме? Если нет → архитектурный.
2. **Пересечение?** Тема файла пересекается с ≥3 существующими тематическими страницами? Если да → архитектурный.
3. **Плотная методология?** Файл выглядит как сквозная модель / протокол (методичка, длинная лекция), а не узкий рецепт/чек-лист? Если да → архитектурный.

Хотя бы одно «да» → архитектурный. Запиши классификацию в рабочий план, формат:
```
[arch] vault/raw/methodologies/<имя>.md — причина: новая тема X
[норм] vault/raw/methodologies/<имя>.md
[arch] vault/raw/lessons/<имя>.md — причина: плотная методология + пересечение с conditions/X
```

**Покажи пользователю классификацию** кратким списком. Спроси: «Подтверди разметку или есть что переклассифицировать?» — **жди ответа.** Это второй и последний gate перед запуском subagents.

Для архитектурных файлов оркестратор также заранее подбирает **контекст ближайших 3-5 тематических страниц**, которые будут переданы в subagent через блок `<ARCHITECTURAL_NEIGHBOUR_PAGES>` (см. Шаг 2).

---

## Шаг 1: подготовка staging

1. Создай `vault/wiki/_staging/` если её нет.
2. Если в `_staging/` уже что-то лежит от прошлого неоконченного прогона — покажи пользователю и спроси: «доделать (Фаза 2 по старым файлам) / удалить и начать заново / отменить». Жди ответа.

## Шаг 2: ФАЗА 1 — параллельная обработка (subagents)

**Модель subagents:** та, которая задана пользователем в `$ARGUMENTS` (`sonnet` или `opus`). Применяется ко всем subagents партии независимо от triage — архитектурные и обычные гоняются на одной модели.

**Размер батча:**
- `$ARGUMENTS == sonnet` → батчи по 4 параллельных.
- `$ARGUMENTS == opus` → батчи по 2 параллельных (дороже, контекст тяжелее).

Сначала пусти архитектурные (их результат важнее получить раньше — задаёт тон таксономии), потом обычные.

**Промпт для subagent (общий шаблон — параметризуется):**

```
Ты — subagent для ingest одного источника в LLM-wiki проекта ЖКТ-курса.

**Источник:** <абсолютный путь>
**Тип:** <methodology | lesson | qna | checklist | article>
**Архитектурный:** <true | false>
**Твоя модель:** <sonnet-4-6 | opus-4-7>

**Режим работы — extended thinking с предмаксимальным усилием.** Это не формальность. Triage архитектурных связок, подбор дословных цитат, архитектурное обоснование, решения по таксономии — всё это задачи, где качество размышлений напрямую определяет результат. Думай развёрнуто перед каждой секцией.

**Твоя задача — ТОЛЬКО подготовить staging-файл.** Не трогай `vault/wiki/sources/`, `index.md`, `log.md`, `conflicts.md`, тематические страницы. Это сделает оркестратор в Фазе 2.

**Что делать:**

1. Прочитай `CLAUDE.md` в корне проекта — там правила проекта (язык русский, авторский стиль, структура vault).

2. Приведи источник к читаемому .md:
   - .pdf → `tools/ingest_pdf.py "<pdf>"` (если рядом нет готового .md с тем же именем)
   - .mp4/.mp3/.wav/.m4a/.webm → `tools/ingest_audio.sh <file> <basename>`; после успешной транскрипции оригинал оставь, удалит оркестратор
   - .md → читай как есть

3. Прочитай полученный .md **целиком** (или по сегментам 20-30 мин для длинных эфиров — feedback_ingest_chunking).

4. [ТОЛЬКО ЕСЛИ АРХИТЕКТУРНЫЙ] Прочитай переданные оркестратором ближайшие тематические страницы:
<ARCHITECTURAL_NEIGHBOUR_PAGES>
<список 3-5 путей к файлам vault/wiki/<папка>/<страница>.md с 1-строчным описанием каждой>
</ARCHITECTURAL_NEIGHBOUR_PAGES>
   Это даст тебе понимание как уже сформулированы соседние страницы и куда вписаться без дубликата/конфликта.

5. **Архитектурные паттерны wiki.** Этот проект следует этим паттернам связок страниц:
   a) **concept + intervention** — теоретическая модель + прикладной протокол (например «ферментация» → «FODMAP-диета»).
   b) **condition + intervention** — состояние/диагноз + способ работы (например «СРК» → «FODMAP-диета»).
   c) **anatomy + concept** — орган + связанное понятие.
   d) **food-group + nutrient** — группа продуктов + нутриентный состав.
   e) **source → 2+ тематических страниц разного типа** — один источник почти всегда ложится на связку (concept+intervention или condition+intervention), не на одну изолированную страницу.

   [ТОЛЬКО ЕСЛИ АРХИТЕКТУРНЫЙ] В секции «Архитектурное обоснование» staging-файла ты ОБЯЗАН явно аргументировать: «этот источник соответствует паттерну X, потому что даёт Y (concept/condition) и Z (intervention/nutrient)». Не допускается «создал страницу X потому что упоминается».

6. Сгенерируй slug: транслит + дефисы, строчные, без пробелов. Пример: «Питание при панкреатите» → `pitanie-pri-pankreatite`.

7. Запиши staging-файл `vault/wiki/_staging/<slug>.md` строго по формату:

   ```markdown
   ---
   staging: true
   raw-file: vault/raw/<путь к исходнику как в файле>
   processed-file: vault/raw/<путь к .md если отличается>
   source-type: <methodology|lesson|qna|checklist|article>
   slug: <slug>
   model: <sonnet-4-6 | opus-4-7>
   architectural: <true | false>
   date-processed: <сегодня YYYY-MM-DD>
   ---

   # <Человекочитаемое название>

   ## Краткое содержание
   2-4 абзаца в духе автора. Не академический пересказ — авторские формулировки и интонация (правило №4 из CLAUDE.md). Частичные цитаты приветствуются.

   ## Ключевые тезисы
   - Тезис 1 (с цитатой если уместно)
   - Тезис 2
   - ... 5-10 пунктов

   ## Цитаты
   > Дословные фрагменты для контента.

   **ПРАВИЛО О ЦИТАТАХ (блокирующее):**
   - Только дословные фрагменты из источника. Никакого пересказа в blockquote.
   - Атрибуции типа «(стр. N)», «(страница N)», «(p. N)», «(MM:SS)» разрешены ТОЛЬКО если они реально присутствуют в исходном .md или объективно выводимы (например Docling сохранил нумерацию страниц, видео-транскрипт с таймкодами). Если источник — markdown-транскрипт без пагинации — атрибуции страниц ЗАПРЕЩЕНЫ. Если в raw нет таймкодов — таймкоды ЗАПРЕЩЕНЫ.
   - Сомневаешься — не пиши атрибуцию. Оркестратор запустит валидатор цитат (`tools/validate_quotes.py`) и отклонит staging-файл если найдутся фабрикации.

   ## [ТОЛЬКО ЕСЛИ АРХИТЕКТУРНЫЙ] Архитектурное обоснование
   1-2 абзаца. Какие тематические страницы создавать/обновлять и какому паттерну связки они соответствуют. Явная форма: «этот источник соответствует паттерну X, потому что...».

   ## Предложения для тематических страниц
   Список concept/condition/symptom/intervention которые стоит создать или обновить. Формат:
   - UPDATE [[conditions/СРК]]: дописать тезис «...»
   - NEW [[interventions/Безглютеновая диета]]: нужна страница, автор даёт подробный протокол
   - UPDATE [[anatomy/Желчный пузырь]]: добавить упоминание

   Не создавай сами страницы. Только предложения — оркестратор решит финально.

   ## Предложения по таксономии
   Если видишь что нужна новая папка в `vault/wiki/` (например `recipes/`, `red-flags/`, `pain-points/`) — обоснуй 1-2 строками. Если изменений не нужно — напиши «без изменений».

   ## Потенциальные конфликты
   Если тезис источника противоречит тому, что уже написано в wiki (ты прочитал соседние страницы на шаге 4, или обычный subagent — прочти те тематические страницы, которые упоминаешь в «Предложениях»), запиши сюда. Формат:
   - [[conditions/X]]: автор говорит A, в wiki написано B. Источник противоречия — стр N / таймкод M (только если реально есть в raw).

   Если конфликтов не нашёл — «не обнаружено».
   ```

8. Верни в ответе: `slug`, путь staging-файла, размер обработанного источника (строк/минут), 1 строку «что в источнике», модель и architectural-флаг.

**Ограничения:**
- Язык только русский.
- НЕ пиши в `sources/`, `index.md`, `log.md`, `conflicts.md`, тематические страницы. Только в `_staging/<slug>.md`.
- НЕ удаляй ничего в `vault/raw/` — оркестратор решит про оригиналы в Фазе 2.
- НЕ редактируй уже существующие файлы в `_staging/` — только создавай свой.
- Сфабрикованные атрибуции цитат (несуществующие страницы/таймкоды) — блокирующая ошибка, оркестратор отклонит.
```

**Подготовка контекста для архитектурных subagents** (делает оркестратор до спавна):
Для каждого архитектурного файла оркестратор выбирает 3-5 ближайших тематических страниц по темам из первых 50 строк raw-файла и имени + индекса wiki. Пути этих страниц подставляются в блок `<ARCHITECTURAL_NEIGHBOUR_PAGES>` в промпте. Это обязательно — без этого subagent работает с плоским index.md.

После каждого батча — кратко сообщи пользователю прогресс: «Батч K/N (модель <model>): готово M staging-файлов».

Если subagent упал (ошибка конвертации, etc) — не останавливай весь прогон, запиши файл в список проблемных и продолжи. В конце Фазы 1 покажи пользователю кто упал.

---

## Шаг 3: ФАЗА 2 — интеграция (оркестратор, т.е. ты)

Теперь работаешь ты сам, без subagents. Работай в том же режиме extended thinking с предмаксимальным усилием — здесь ты принимаешь финальные решения по таксономии, связкам страниц и разрешению цитатных проблем.

### 3.1 Прочитай все staging-файлы

`vault/wiki/_staging/*.md`. Получи картину всей партии целиком.

### 3.2 Валидация цитат (обязательный gate)

Для каждого staging-файла запусти:

```bash
python3 tools/validate_quotes.py vault/wiki/_staging/<slug>.md <raw .md path из frontmatter>
```

Обработка результатов:
- **exit 0** — идёшь дальше с этим файлом.
- **exit 1** — найдены галлюцинации или сфабрикованные атрибуции. Действия:
  1. Прочитай вывод.
  2. Для каждой проблемы: либо замени цитату на настоящую из raw.md (substring-match), либо удали цитату, либо убери фабрикованную атрибуцию.
  3. Перезапусти валидатор. Повтори до 2 итераций.
  4. Если не чистится — помечай секцию «Цитаты» в staging-файле пометкой `<!-- TODO: ручная ревизия цитат, валидатор упал -->` и продолжай интеграцию (оркестратор зафиксирует это в log.md).

Веди подсчёт: сколько staging-файлов прошли сразу, сколько после правок, сколько ушли с пометкой TODO.

### 3.3 Таксономические решения — единым пакетом

Собери все секции «Предложения по таксономии» из staging-файлов. Если есть предложения новых папок — покажи пользователю одним списком: «Предложения по таксономии от этой партии: [...]. Что принимаем?» Жди ответа. Если без изменений — иди дальше.

### 3.4 Интеграция staging → wiki

Для каждого staging-файла (последовательно, не параллельно в этой фазе — из-за shared file conflicts):

- Создай `vault/wiki/sources/<slug>.md` по формату из `CLAUDE.md`:
  - frontmatter: добавь `architectural: <значение из staging>`;
  - секции: Краткое содержание, Ключевые тезисы, Связанные страницы, **Архитектурное обоснование** (только если `architectural: true`), Цитаты.
- Перенеси контент из staging + добавь wiki-links в «Связанные страницы» на базе секции «Предложения для тематических страниц».
- Создай/обнови тематические страницы по решённой таксономии:
  - для архитектурных источников — строго по паттерну связок из Архитектурного обоснования (не создавай изолированные страницы);
  - в существующие дописывай секцию «Из [[sources/<slug>]]: ...» — не переписывай старое.
- Конфликты из секции «Потенциальные конфликты» → записью в `vault/wiki/conflicts.md` (не разрешай на месте).

### 3.5 Обнови `vault/wiki/index.md`

Добавь все новые страницы разом в соответствующие группы.

### 3.6 Обнови `vault/wiki/log.md`

По одной записи на каждый ingested источник (формат из `CLAUDE.md`):

```markdown
## [YYYY-MM-DD] ingest | <название>
- Источник: `vault/raw/<путь>`
- Модель: <sonnet-4-6 | opus-4-7>    # та, что была задана в $ARGUMENTS
- Triage: архитектурный | обычный
- Создана страница: [[sources/<slug>]]
- Обновлены: [[...]]
- Новые тематические страницы: [[...]]
- Архитектурный паттерн: concept+intervention | ... | нет
- Валидация цитат: ok | ok после N итераций | TODO ручная ревизия
- Новые расхождения в conflicts.md: <кратко> | нет
- Таксономия: фактическое решение из шага 3.3
- Заметки: ...
```

### 3.7 Удали оригиналы

Если применимо по правилу стандарта аудио/видео (см. CLAUDE.md, раздел «Работа с тяжёлыми источниками»): после успешной транскрипции оригинальный .mp4/.m4a/.wav удаляется, остаётся .mp3 + .md. Перед удалением — покажи пользователю список что удаляешь. Жди подтверждения.

### 3.8 Очисти `vault/wiki/_staging/`

Удали все обработанные staging-файлы. Оставь файлы с пометкой TODO-валидации (если такие были) и сообщи пользователю.

### 3.9 Финальный отчёт

- Обработано: N источников (arch: K, обычных: N-K)
- Модель партии: <sonnet-4-6 | opus-4-7>
- Валидация цитат: ok X, ok после правок Y, TODO ручная ревизия Z
- Созданы страницы-источники: [...]
- Созданы тематические страницы: [...]
- Обновлены тематические страницы: [...]
- Таксономия: <что изменилось или «не менялась»>
- Конфликты: M записей добавлено в conflicts.md
- Проблемные файлы Фазы 1 (если есть): [...]

---

## Напоминания

- Язык только русский.
- Авторский стиль — и в staging, и в финальных страницах.
- **Модель задаётся через `$ARGUMENTS` — `sonnet` или `opus`.** Нет параметра — не запускайся.
- **Haiku не используется на любом шаге этого workflow.**
- Обе допустимые модели работают в режиме extended thinking с предмаксимальным усилием — и оркестратор, и subagents.
