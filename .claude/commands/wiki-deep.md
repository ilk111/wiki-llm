# Query по графу wiki — deep mode

Пользователь спрашивает: **$ARGUMENTS**

Ответь **только на основе `vault/wiki/`** — но в этот раз с максимально полным обходом графа. Не используй общие знания. Если в wiki данных не хватает — покажи конкретно, чего и где не хватает (orphan-темы, pending-конфликты).

## Процедура обхода (2-3 хопа + sources + conflicts)

1. **Контекст сессии.** Быстро просмотри последние 20-30 строк `vault/wiki/log.md` — пойми, что ингестнуто недавно, это может повлиять на ответ.

2. **Точка входа — `vault/wiki/index.md`.** Прочитай целиком. Выпиши все страницы, потенциально релевантные вопросу (широкий bucket, будем фильтровать по ходу).

3. **Первый хоп.** Прочитай отобранные страницы. Из frontmatter извлеки `sources:`, из тела — все `[[wiki-links]]`.

4. **Второй хоп.** Пройди по собранным `[[ссылкам]]`. Отбрасывай только то, что явно не относится к вопросу.

5. **Третий хоп — по необходимости.** Если на втором хопе всплывают ссылки на дополнительные концепты, которые нужны для полноты ответа (механизм, смежная интервенция, тест и т.д.) — сходи и туда. Остановись, когда новые страницы не добавляют информации к ответу.

6. **Прочитай релевантные `vault/wiki/sources/<slug>.md`.** Цитаты из источников — ключевая ценность. Бери дословные фрагменты для ответа (в духе правила №4 из CLAUDE.md — сохраняй авторский голос).

7. **Проверь `vault/wiki/conflicts.md`.** Если по теме вопроса есть pending-расхождения между источниками — обязательно упомяни их в ответе, не прячь.

8. **`vault/raw/` читай только если** в `sources/<slug>.md` нет нужной цитаты, а она критична для ответа. По умолчанию raw не трогаем.

## Формат ответа

Структурируй ответ под сложность вопроса, но обязательно включи:

- **Основной ответ** — в духе материалов wiki, с сохранением авторских формулировок. Каждый значимый тезис — с пометкой источника: `([[conditions/СРК]], [[sources/lesson-05]])`.

- **Цитаты** (если уместно) — дословные фрагменты из `sources/` с указанием slug'а источника.

- **Конфликты источников** — если по теме есть записи в `conflicts.md`, покажи их явно («источник A говорит X, источник B — Y, не разрешено»).

- **Пробелы в wiki** — если обход показал, что чего-то не хватает (концепт упоминается, но страницы нет; связь очевидна, но `[[ссылки]]` нет; тема orphan) — перечисли их в конце блоком «Что стоит дозаполнить».

- **Trail обхода** — список всех прочитанных страниц в порядке посещения, с пометкой хопа (h1/h2/h3/source). Чтобы ты мог проверить, что я не пропустил нужное и не читал лишнего.

## Правила файловых операций

- **Не создавай и не редактируй страницы wiki молча.** Если ответ тянет на новую страницу или заслуживает допиcки в существующую — в конце предложи конкретный план правок и жди подтверждения.

- **Допиши запись в `vault/wiki/log.md`** по формату `## [YYYY-MM-DD] query | <короткое описание>` (см. CLAUDE.md) — deep-запросы стоят того, чтобы их логировать.

- **Если пользователь скажет «зафайль»** — создай новую страницу в подходящей папке `vault/wiki/` по формату тематической страницы из CLAUDE.md, обнови `index.md`, отметь в `log.md`.

## Если вопрос не требует deep-режима

Если после прочтения index.md видно, что ответ очевидно помещается в 1-2 страницы — скажи: «Вопрос не требует deep-обхода, хватит `/wiki $ARGUMENTS`». Не раздувай ответ искусственно.
