# Ingest методичек в wiki

Запусти workflow ingest по правилам из `CLAUDE.md` (раздел «Workflow: Ingest»).

Модель выбирает пользователь (какую подключил в IDE — ту и используешь). Работай в режиме extended thinking с предмаксимальным усилием на протяжении всего workflow: triage, архитектурное обоснование, подбор цитат, решения по таксономии — это всё задачи, где качество размышлений напрямую определяет результат.

## Шаг 0: найди новые файлы

1. Прочитай `vault/wiki/log.md` и `vault/wiki/index.md` — пойми, что уже ингестнуто.
2. Посмотри `vault/raw/methodologies/` (Glob/ls).
3. Сравни: какие `.pdf`/`.md` там **ещё не** имеют соответствующей страницы в `vault/wiki/sources/` и не упомянуты в `log.md`.
4. Покажи пользователю список неингестнутых методичек, спроси — какую (или все) обрабатываем. Жди ответа.

## Для каждой выбранной методички

### 1. Определи формат и приведи к .md

- Если рядом с `.pdf` уже лежит `.md` с тем же названием — читай `.md` (уже конвертировано).
- Если только `.pdf` — прогони `tools/ingest_pdf.py "<pdf>"`, получи `.md` рядом.
- Если аудио/видео — `tools/ingest_audio.sh <file> <basename>`, оригинал не удаляй до конца ingest.

### 2. Прочитай источник целиком (Read)

Для длинных источников (>30 мин речи) — по сегментам 20-30 мин, но **весь**.

### 3. Triage: архитектурно значимый или обычный?

Ответь на 3 вопроса письменно (коротко, в мыслях или в комментарии):
1. **Новая тема?** Есть ли в `index.md` уже ≥2 страницы по этой теме? Если нет — это первый источник в разделе → архитектурный.
2. **Пересечение?** Упоминает ли источник ≥3 тем, по которым уже есть страницы в wiki? Если да — высокий риск конфликтов, нужна интеграция → архитектурный.
3. **Плотная методология?** Даёт ли источник сквозную модель (не просто чек-лист), которую нужно разложить на concept + intervention или condition + intervention? Если да → архитектурный.

Если хотя бы одно «да» — источник **архитектурный**, применяй расширенный контекст (шаг 4.Б). Иначе — обычный (шаг 4.А).

### 4. Загрузи контекст wiki

**4.А (обычный источник):** Прочитай `vault/wiki/index.md` целиком. Этого достаточно.

**4.Б (архитектурный источник):** Прочитай:
- `vault/wiki/index.md` целиком;
- **3-5 ближайших тематических страниц**, которые уже есть в wiki и касаются темы источника — не только заголовки, **содержимое**;
- последние 10-15 записей из `vault/wiki/log.md` — чтобы видеть контекст недавних решений по таксономии;
- `vault/wiki/conflicts.md` целиком.

### 5. Архитектурные паттерны wiki (обязательная секция для Opus и архитектурных источников)

Существующие паттерны связок, которые держат скелет wiki:
1. **concept + intervention** — теоретическая модель + прикладной протокол (например «ферментация» → «FODMAP-диета»).
2. **condition + intervention** — состояние/диагноз + способ работы с ним (например «СРК» → «FODMAP-диета»).
3. **anatomy + concept** — орган + связанное понятие (например «желчный пузырь» → «отток желчи»).
4. **food-group + nutrient** — группа продуктов + её нутриентный состав.
5. **source → 2+ тематических страниц разного типа** — один источник почти всегда питает связку concept+intervention или condition+intervention, а не одну изолированную страницу.

**Для архитектурных источников:** в самой странице-источнике (шаг 7) будет секция «Архитектурное обоснование», где ты явно аргументируешь: «этот источник ложится на паттерн X, потому что даёт Y на уровне concept и Z на уровне intervention». Не допускается «создал страницу X потому что упоминается».

### 6. Суммаризация тезисов (без ожидания)

5-10 пунктов «что важного для курса/контента». Сохраняй авторский стиль, частичные цитаты (правило №4). Покажи пользователю, но **не жди реакции** — иди дальше. Если пользователь поправит, откатишь.

### 7. Создай страницу-источник

`vault/wiki/sources/<slug>.md`:

```yaml
---
type: source
source-type: methodology | lesson | qna | checklist | article
date-ingested: YYYY-MM-DD
raw-file: ../../raw/<категория>/<имя>
architectural: true | false    # результат triage из шага 3
---
```

Секции: Краткое содержание → Ключевые тезисы → Связанные страницы → **Архитектурное обоснование** (только если `architectural: true`) → Цитаты.

**Архитектурное обоснование** (1-2 абзаца): какие тематические страницы создаются/обновляются и какому паттерну связки они соответствуют. Явная аргументация в формате «этот источник соответствует паттерну X, потому что...».

**Цитаты** — только дословные фрагменты из источника. **Запрещено** выдумывать атрибуции, которых нет в исходнике:
- если источник — `.md` транскрипт без страниц/таймкодов → в цитате не должно быть «(стр. N)», «(p. N)», «(YY:MM)»;
- если источник — `.pdf` без надёжной пагинации после Docling → то же самое;
- если в исходнике есть таймкоды или явные указания страниц — можно ссылаться.

Валидация этого правила — автоматически на шаге 10.

### 8. Расхождения с уже ингестнутой wiki

Если тезис противоречит существующей странице — не переписывай её на месте. Добавь запись в `vault/wiki/conflicts.md` с обоими источниками и пометкой «к разрешению».

### 9. Обнови тематические страницы

- Новые concept/condition/symptom/intervention — создай по формату из `CLAUDE.md`.
- Существующие — допиши в «Что говорят источники» ссылку `[[sources/<slug>]]` и тезисы.
- Wiki-links `[[Имя]]`, не markdown-links.
- **Не переписывай противоречащие места** — они уходят в `conflicts.md` (шаг 8).
- Для архитектурного источника — создавай тематические страницы по паттернам из шага 5, не изолированными вкраплениями.

### 10. Валидация цитат (обязательный gate)

Запусти:

```bash
python3 tools/validate_quotes.py vault/wiki/sources/<slug>.md <путь к raw .md>
```

Интерпретация:
- exit 0 → цитаты в порядке, идёшь дальше.
- exit 1 → найдены галлюцинации или сфабрикованные атрибуции (страницы/таймкоды). Действия:
  1. Прочитай вывод валидатора.
  2. Для каждой проблемной цитаты: либо найди в raw.md настоящую формулировку и замени, либо удали цитату, либо убери фабрикованную атрибуцию.
  3. Перезапусти валидатор. Повтори пока не станет exit 0.
  4. Если после 2 итераций всё ещё не чистится — помечай секцию «Цитаты» пометкой `<!-- TODO: ручная ревизия цитат -->` и пиши об этом в `log.md` (шаг 12).

Частичные совпадения (partial) — не блокируют, но требуют беглой проверки. Считаются ок.

### 11. Обнови `vault/wiki/index.md`

Добавь новые страницы в соответствующие группы.

### 12. Допиши запись в `vault/wiki/log.md`

```markdown
## [YYYY-MM-DD] ingest | <название>
- Источник: `vault/raw/<категория>/<файл>`
- Модель: <opus-4-7 | sonnet-4-6>
- Triage: архитектурный | обычный (причина 1 строкой)
- Создана страница: [[sources/<slug>]]
- Обновлены: [[...]]
- Новые тематические страницы: [[...]]
- Архитектурный паттерн: concept+intervention | condition+intervention | ... | нет (для архитектурных)
- Валидация цитат: ok | ok после N итераций | TODO ручная ревизия
- Новые расхождения в conflicts.md: <кратко> | нет
- Таксономия: не менялась | предложил X → ответ Y
- Заметки: ...
```

### 13. Оцени таксономию

Правило живой таксономии:
- Нужна ли новая папка / переименование / перераскладка?
- Если да — задай пользователю явный вопрос «Предлагаю `vault/wiki/X/` потому что Y. Согласен?» и **жди ответа**.
- Если нет — напиши в `log.md` «таксономия не менялась».

## Напоминания

- Язык только русский.
- `vault/raw/` — immutable, только читать.
- Пишешь только в `vault/wiki/`.
- Авторский стиль сохраняй (правило №4 из `CLAUDE.md`).
- Haiku для ingest не используется.
- Цитаты — только дословно; сфабрикованные атрибуции (страницы, таймкоды) — блокирующая ошибка, валидатор их ловит.
- Единственные интерактивные gates: (0) выбор источника, (13) таксономия.
- На шаге 6 (тезисы) **не жди** — иди дальше.
