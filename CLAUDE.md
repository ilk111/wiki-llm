# Схема LLM-wiki

Это инструкция для Claude (тебя) — как поддерживать эту wiki. База — паттерн Karpathy `llm-wiki.md` в корне проекта. Прочитай его сначала, если ещё не читал.

**Это шаблон.** При развёртывании новой wiki — замени `{{ПРЕДМЕТ}}` ниже на свою тему (курс, книга, исследование, компания и т.п.) и скорректируй правила под свой язык/домен. Всё остальное — структура, workflow, форматы страниц — универсально.

---

## Предмет wiki

**Тема:** `{{ПРЕДМЕТ}}` — короткая формулировка (1-2 предложения).

**Источники:** `{{ТИПЫ ИСТОЧНИКОВ}}` — лекции, эфиры, методички, статьи, книги, транскрипты и т.д.

**Цель:** `{{ЦЕЛЬ}}` — для чего ведётся (контент-маркетинг, личный research, онбординг команды, подготовка курса...).

---

## Базовые правила

**Правило №1: язык.** Выбери язык wiki и придерживайся его во всех страницах, коммитах, ответах пользователю, подборе источников. Не смешивай языки внутри одной wiki.

**Правило №2: `vault/raw/` — immutable.** Никогда не редактируй файлы там. Только читаешь.

**Правило №3: всё, что пишешь — только в `vault/wiki/`.**

**Правило №4: сохраняй авторский стиль при конспектировании.** Когда делаешь страницу-источник или пересказ в тематических страницах — держи авторские формулировки и интонацию, не переписывай сухо-академически. Финальная цель wiki, как правило, — использовать накопленное знание (для контента, решений, выводов). Авторский голос = ценность. Ключевые тезисы пиши максимально близко к формулировкам автора (частичные цитаты приветствуются), секции «Что говорят источники» — с характерными оборотами. Канцелярит и безличный пересказ запрещены.

**Правило №5: `vault/raw/` исключён из графа Obsidian.** В `vault/.obsidian/graph.json` стоит фильтр `"search": "-path:\"raw/\""` — это скрывает raw-файлы (Docling-выгрузки, транскрипты) из view графа, чтобы они не висели как orphan-ноды без связей. Граф должен отражать только смысловую сетку wiki. Если создаёшь новый подвид raw-материалов (например `raw/social/`) — он автоматически попадает под фильтр. Если добавляешь новые **не**-raw папки в vault (например `vault/drafts/`) — учти, что они будут показываться в графе; при необходимости расширь фильтр в `graph.json`.

## Структура

```
vault/                       # Obsidian vault (открывается как vault в Obsidian)
├── raw/                     # immutable источники
│   ├── lessons/             # уроки / лекции (mp4 + .md транскрипт рядом)
│   ├── qna/                 # эфиры вопрос-ответ / интервью
│   ├── methodologies/       # методички / руководства (pdf + .md)
│   ├── checklists/          # чек-листы
│   ├── articles/            # внешние статьи
│   └── assets/              # картинки, схемы
└── wiki/                    # тут пишешь только ты
    ├── index.md             # каталог всех страниц
    ├── log.md               # хронология ingest/query/lint
    ├── conflicts.md         # накопительный список расхождений между источниками (разрешаются скопом, а не инлайн)
    ├── sources/             # 1 страница на каждый ingested источник
    └── (другие папки появляются по ходу — см. правило живой таксономии)
```

**Категории в `raw/` — это стартовый набор.** Добавляй свои (`raw/podcasts/`, `raw/papers/`, `raw/transcripts/` и т.п.) по мере необходимости. Удаляй неиспользуемые.

## Правило живой таксономии (важно)

В `vault/wiki/` на старте есть только `sources/`. Тематические папки — **не создавай заранее**. Они появляются когда из реального материала становится понятно, как пользователь формулирует эти срезы.

**Примеры возможных тематических папок** (зависит от домена):
- Курс/обучение → `concepts/`, `interventions/`, `conditions/`, `symptoms/`, `mistakes/`, `pain-points/`, `red-flags/`
- Книга/художественное → `characters/`, `places/`, `events/`, `themes/`
- Research → `hypotheses/`, `findings/`, `methods/`, `gaps/`
- Компания → `people/`, `products/`, `decisions/`, `processes/`

**После КАЖДОГО ingested источника:**
1. Оцени: имеет ли смысл изменить таксономию (создать новую папку, переименовать существующую, переразложить страницы)?
2. Если да — задай пользователю явный вопрос: «Предлагаю создать `vault/wiki/X/` потому что Y. Согласен / есть варианты лучше?»
3. Если нет — напиши в `log.md` строкой «таксономия не менялась».

Не действуй молча. Не накапливай долг правок. Если сомневаешься — спрашивай.

## Workflow: Ingest

Когда пользователь указывает на новый источник в `vault/raw/`:

1. **Прочитай источник** целиком (если это .md транскрипт — Read; если pdf — сначала прогони через `tools/ingest_pdf.py`; если видео/аудио — `tools/ingest_audio.sh`).

2. **Суммаризируй ключевые тезисы автономно** — 5-10 пунктов «что важного для цели wiki», с частичными цитатами в духе автора. Покажи пользователю, но **не жди реакции** — иди дальше. Если пользователь захочет скорректировать, он скажет и ты откатишь изменения.

3. **Расхождения с уже ингестнутыми источниками — в `vault/wiki/conflicts.md`.** Если тезис противоречит тому, что уже написано в wiki (по другому источнику), не разрешай на месте — добавляй запись в `conflicts.md` и помечай в тексте страницы-источника. Разногласия разрешаются скопом в отдельной сессии.

4. **Создай страницу-источник** в `vault/wiki/sources/<slug>.md` (см. формат ниже).

5. **Обнови или создай тематические страницы** в соответствующих папках wiki:
   - Если упомянут новый concept/entity/etc., которого ещё нет — создай страницу.
   - Если упомянуто что-то существующее — допиши секцию «упоминается в [[sources/<slug>]]» с ключевыми тезисами оттуда.
   - Используй wiki-links `[[Имя страницы]]` (Obsidian-синтаксис, не markdown links).

6. **Обнови `index.md`** — добавь все новые страницы.

7. **Допиши запись в `log.md`** (формат ниже). Упомяни, если добавил что-то в `conflicts.md`.

8. **Оцени таксономию** (см. правило живой таксономии). Если есть предложение изменить структуру папок — **спроси пользователя и жди ответа** (это единственный интерактивный gate в ingest, кроме выбора источника).

## Формат страницы-источника (`vault/wiki/sources/<slug>.md`)

```markdown
---
type: source
source-type: lesson | qna | methodology | checklist | article
date-ingested: YYYY-MM-DD
raw-file: ../../raw/<категория>/example
---

# <человекочитаемое название>

## Краткое содержание
2-4 абзаца — о чём источник, для кого, какой главный тезис.

## Ключевые тезисы
- Тезис 1 (с цитатой если уместно)
- Тезис 2
- ...

## Связанные страницы
- [[<категория>/<имя>]]
- ...

## Цитаты
> Дословные фрагменты, которые могут пригодиться для цели wiki — с пометкой откуда (тайм-код для видео, страница для pdf если есть).
```

## Формат тематической страницы

```markdown
---
type: <condition | symptom | concept | intervention | character | place | ...>
sources: [[sources/<slug-1>]], [[sources/<slug-2>]]
last-updated: YYYY-MM-DD
---

# <Название>

## Что это
Короткое определение.

## Механизм / физиология / контекст
(если применимо)

## Что говорят источники
- Из [[sources/<slug>]]: основной тезис
- Из [[sources/<другой slug>]]: уточнение / противоречие

## Связанные
- [[<страница>]]
- ...

## Углы для использования
(когда применимо — потенциальные форматы, применения, выводы)
```

## Формат `index.md`

Группировка по папкам wiki. Каждая страница — одна строка: `- [[путь/страница]] — однострочное описание`.

## Формат `log.md`

Append-only. Каждая запись начинается с заголовка фиксированного формата для grep'абельности:

```markdown
## [YYYY-MM-DD] ingest | <название источника>
- Источник: `vault/raw/<путь>`
- Создана страница: [[sources/<slug>]]
- Обновлены: [[<страница1>]], [[<страница2>]]
- Новые тематические страницы: [[<страница>]]
- Таксономия: не менялась | предложил пользователю X → ответ Y
- Заметки: ...

## [YYYY-MM-DD] query | <короткое описание вопроса>
- Использованные страницы: ...
- Результат: ...
- Зафайлено в wiki: [[<страница>]] | нет

## [YYYY-MM-DD] lint
- Найдено: ...
- Исправлено: ...
```

## Workflow: Query

Когда пользователь задаёт вопрос:

1. Прочитай `vault/wiki/index.md` — найди релевантные страницы.
2. Прочитай эти страницы.
3. Если данных не хватает — прочитай нужные источники из `vault/wiki/sources/` или связанные тематические страницы.
4. Дай ответ с цитированием wiki-страниц.
5. Спроси: «Зафайлить ответ как новую страницу в wiki?» Если да — создай `vault/wiki/<подходящая папка>/<slug>.md` и допиши `log.md`.

Быстрый обход в 1 хоп — `/wiki <вопрос>`. Глубокий обход в 2-3 хопа + sources + conflicts — `/wiki-deep <вопрос>`.

## Workflow: Lint

По запросу пользователя («залинтуй wiki»):

- Найди противоречия между страницами (одно говорит X, другое not-X).
- Найди устаревшие утверждения (новые источники их опровергают).
- Найди orphan-страницы (нет inbound `[[ссылок]]`).
- Найди концепции, упомянутые в нескольких местах но без своей страницы.
- Найди недостающие cross-references.
- Сформируй отчёт пользователю + предложи план правок. Не правь сам молча.

## Инструменты

В `tools/`:
- `extract_audio.sh <input> [out.mp3]` — перекодировать в mp3 mono 128kbps 44100Hz (стандарт wiki для речи).
- `transcribe.py <audio> [out.md]` — Whisper API транскрипция (русский по умолчанию, поменяй `language=` в скрипте под свой язык; авточанкинг под 25MB лимит).
- `ingest_pdf.py <pdf> [out.md]` — Docling PDF → markdown.
- `ingest_audio.sh <input> [out_basename]` — универсальный: аудио/видео → .mp3 (переслушивание) + .md (транскрипт).
- `ingest_video.sh <video> [out.md]` — видео → транскрипт в один шаг (mp3 не сохраняется).
- `validate_quotes.py <result.md> <raw.md>` — проверяет что все blockquote-цитаты реально есть в raw; ловит фабрикации атрибуций (страниц, таймкодов).

## Работа с тяжёлыми источниками (аудио/видео)

**Стандарт:** держим в `vault/raw/` только перекодированный `.mp3` (mono 128kbps 44100Hz) + `.md` транскрипт. Оригиналы (.m4a, .mp4, .wav и т.п.) **не храним** — после успешной транскрипции оригинал удаляется.

**Процедура ingest:**
1. Положить оригинальный файл в соответствующую папку `vault/raw/<категория>/`.
2. Запустить `tools/ingest_audio.sh <путь к оригиналу> <путь без расширения>` — создаст `.mp3` + `.md` с нужным slug-именем.
3. Бегло проверить транскрипт (начало/конец, длина разумна).
4. Удалить оригинал: `rm <путь к оригиналу>` (скрипт печатает готовую команду в конце).
5. Дальше — обычный ingest-workflow (страница-источник, тематические страницы, log.md, таксономия).

**Почему 128k 44100 mono:** Whisper и так ресэмплит, но mp3 нужен не только для ASR — мы его храним для переслушивания (Obsidian играет `.mp3` рядом с `.md`), поэтому качество должно быть «человеческим». 128k mono ≈ 57 MB/час — компромисс между разборчивостью и объёмом.

Перед запуском: `tools/.env` должен содержать `OPENAI_API_KEY`. Виртуальное окружение в `tools/.venv` (см. `README.md` в корне шаблона).

## Контент-ориентация / use-case ориентация

Финальная цель wiki — не просто хранилище, а рабочий инструмент для цели, которую ты заявил выше. Поэтому:
- Источники, где люди задают вопросы своими словами (эфиры Q&A, форумы, интервью), — источник #1 для понимания реальных формулировок твоей целевой аудитории.
- Каждая тематическая страница со временем должна обрасти секцией «углы для использования» (посты, сценарии, решения, выводы — что именно ты будешь делать из накопленного).
- При query'ях вида «дай мне идеи / варианты / выводы по X» — компилируй из wiki, ничего не выдумывай.

## Что делать в начале каждой сессии

1. Прочитай `vault/wiki/log.md` (последние записи) — поймёшь что сделано недавно.
2. Прочитай `vault/wiki/index.md` — узнаешь актуальное состояние.
3. Если структура wiki непонятна — пробегись по `vault/wiki/` через Glob/ls.
